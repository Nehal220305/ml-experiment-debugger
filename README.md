# 🔬 ML Experiment Debugger Environment

> **The only OpenEnv environment where training logs are REAL — genuine PyTorch output, not simulated.**

An OpenEnv RL environment where an AI agent debugs broken ML training experiments.
The agent receives a real faulty training log, identifies the bug, and proposes a fix.

**Live API:** https://Nehal-2203-ml-experiment-debugger.hf.space/docs

---

## Quick Start

```python
import requests

BASE_URL = "https://Nehal-2203-ml-experiment-debugger.hf.space"

# Reset — get a real broken PyTorch training log
obs = requests.post(f"{BASE_URL}/reset", json={"task_id": "easy"}).json()
print(obs["observation"]["training_log"][:3])
# ["Training config: lr=100.0, optimizer=sgd, loss=bce",
#  "step 1: loss=0.745 grad_norm=4.30e-01",
#  "step 4: loss=nan (EXPLODING GRADIENTS, grad_norm=1.59e+06)"]

# Diagnose — submit free-text analysis, get scored
result = requests.post(f"{BASE_URL}/step", json={"action": {
    "action_type": "diagnose",
    "response": "Learning rate 100.0 is too high causing gradient explosion and NaN loss. Fix: reduce learning rate to 0.01."
}}).json()
print(f"Score: {result['reward']:.2f}")
# Score: 0.90
```

## What is this?

Real ML engineers debug broken training runs every day — exploding loss,
data leakage, silent label corruption, vanishing gradients. This environment
tests those scenarios using **real PyTorch training loops** so AI agents can
learn to diagnose and fix them programmatically.

Unlike other OpenEnv environments that simulate responses, every training log
in ML Experiment Debugger is generated by **real PyTorch code** — gradient norms,
loss explosions, and vanishing signals are mathematically genuine.

## Tasks

| Task | Bug | Difficulty | Description |
|------|-----|------------|-------------|
| easy | `learning_rate_too_high` | Easy | Loss explodes to NaN — real gradient explosion via PyTorch |
| medium | `data_leakage` | Medium | Val set == train set, accuracy misleadingly perfect |
| hard | `label_noise` | Hard | 30% labels flipped + bad LR, model appears fine but fails on real data |
| very_hard | `wrong_loss_function` | Very Hard | MSE loss for classification — accuracy plateaus near 52% despite converging loss |
| expert_1 | `vanishing_gradients` | Expert | Deep network with sigmoid activations — gradients vanish, loss stalls |
| expert_2 | `missing_normalization` | Expert | Unnormalized inputs (values >1000) cause wildly oscillating loss |

## Action Space

The agent sends a JSON action with a free-text diagnosis:
```json
{
  "action_type": "diagnose",
  "response": "The learning rate of 50.0 is too high causing gradient explosion. The loss explodes to NaN within 5 steps. Fix: reduce learning rate to 0.01."
}
```

### Action types

- `diagnose` — agent provides free-text diagnosis and fix. Scored by LLM-as-judge on bug identification, fix quality, and explanation depth.
- `request_more_steps` — agent requests 10 more training steps to observe patterns before diagnosing.
- `inspect_gradients` — agent requests gradient norm statistics to identify gradient-related bugs.

## Example Agent Interaction

### 1. Reset — start a new episode
```bash
POST /reset {"task_id": "easy"}
```
```json
{
  "training_log": [
    "Training config: lr=100.0, optimizer=sgd, loss=bce",
    "step 1: loss=0.745 train_acc=0.306 val_acc=0.262 grad_norm=4.30e-01",
    "step 2: loss=63.52 train_acc=0.641 val_acc=0.625 grad_norm=2.56e+01",
    "step 4: loss=nan (EXPLODING GRADIENTS, grad_norm=1.59e+06)"
  ],
  "current_config": {"learning_rate": 100.0, "max_iter": 10, "optimizer": "sgd"},
  "message": "Task 'easy': Training loss explodes to NaN. Diagnose and fix the config."
}
```

### 2. Inspect gradients (optional multi-turn step)
```bash
POST /step {"action": {"action_type": "inspect_gradients"}}
```
```json
{
  "training_log": [
    "step 1: loss=0.745 grad_norm=4.30e-01",
    "step 2: loss=63.52 grad_norm=2.56e+01",
    "step 4: loss=nan grad_norm=1.59e+06"
  ],
  "message": "Gradient inspection results. Use this to identify gradient-related bugs."
}
```

### 3. Submit diagnosis — get scored
```bash
POST /step {"action": {"action_type": "diagnose", "response": "The learning rate of 100.0 is too high causing exploding gradients. Gradient norm exceeds 1e6 by step 4. Fix: reduce learning rate to 0.01."}}
```
```json
{
  "reward": 0.95,
  "done": true,
  "feedback": "✓ Correctly identified bug | ✓ Suggested valid fix | ✓ Good explanation | [LLM-judge: correct diagnosis]",
  "message": "Score: 0.95. Episode complete."
}
```

## Why This Environment Matters

Debugging broken ML experiments is a task every ML engineer faces daily.
No existing OpenEnv environment covers this domain. This environment enables:

- Training RL agents to diagnose ML failures automatically
- Evaluating LLMs on realistic engineering debugging tasks
- Benchmarking agent performance across 6 difficulty levels with calibrated caps
- Testing agent ability to reason about gradient flow, data quality, and loss design

> Unlike simulated environments, every observation here is computed by **running real PyTorch code** — making this a genuine test of ML reasoning, not pattern matching on fake logs.

---

## Environment Details

- **Framework:** OpenEnv + FastAPI + Docker
- **Grader:** LLM-as-judge (Llama-3.3-70B) with keyword fallback — evaluates bug identification, fix quality, and explanation depth
- **Training:** Real inline PyTorch execution — all loss curves and metrics are genuine PyTorch output
- **Randomized:** Bug parameters vary per episode — agents cannot memorize answers
- **Multi-turn:** Agents can request more steps or inspect gradients before diagnosing
- **Multi-agent:** True concurrent session isolation per episode_id
- **Session Management:** Auto-expiry of abandoned sessions after 1 hour
- **Live Space:** https://huggingface.co/spaces/Nehal-2203/ml-experiment-debugger
